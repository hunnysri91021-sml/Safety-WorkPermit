# SML Work Permit System

ระบบขออนุญาตปฏิบัติงาน (Work Permit) สำหรับบริษัท สยามกลการโลจิสติกส์ จำกัด
ทำงานบน Google Sheets + Google Apps Script (ฟรี ไม่ต้องมี license เพิ่ม) และ
โฮสต์หน้าเว็บผ่าน GitHub Pages

## แผนผังไฟล์

| ไฟล์ | ใช้โดย | หน้าที่ |
|---|---|---|
| `index.html` | ทุกคน | หน้าแรก/เมนูเลือกใช้งาน |
| `request-permit.html` | ผู้รับเหมา | ฟอร์มยื่นขอ Work Permit |
| `view.html` | ทุกคน (ผ่าน QR) / รปภ. | ตรวจสอบสถานะ Permit + ปุ่มบันทึกเข้า-ออกสำหรับ รปภ. |
| `quiz.html` | ผู้รับเหมา | แบบทดสอบความปลอดภัย (เฉพาะงานเสี่ยงสูง) ก่อนออกบัตร |
| `guard-login.html` | รปภ. | เข้าสู่ระบบด้วย PIN เพื่อใช้งาน `view.html` โหมด รปภ. |
| `site-report.html` | ผู้รับเหมา | ส่งรูปประเมินพื้นที่หลังเข้างาน |
| `dashboard.html` | Safety Team | สรุปข้อมูล realtime (ล็อกด้วยรหัสผ่าน) |
| `WorkPermit-AppsScript.gs` | Safety Team (ติดตั้ง) | Backend ทั้งหมด — วางใน Google Apps Script ที่ผูกกับ Sheet |

ทุกหน้า `.html` ต้องตั้งค่าคงที่ `GOOGLE_SCRIPT_URL` ให้ตรงกับ URL ของ Web App
ที่ deploy จาก `WorkPermit-AppsScript.gs` (ยกเว้น `index.html` ที่เป็นแค่เมนู)

## ข้อมูลส่วนบุคคลและ PDPA

ระบบนี้เก็บ **ข้อมูลส่วนบุคคล** ของผู้รับเหมา/ผู้ปฏิบัติงาน และของ รปภ. โดยตรง
จึงต้องอยู่ภายใต้ พ.ร.บ. คุ้มครองข้อมูลส่วนบุคคล (PDPA) ทุกครั้งที่แก้ไขโค้ดหรือ
เพิ่มฟีเจอร์ใหม่ ให้ตรวจทาน 4 ข้อนี้เสมอ:

### 1. เก็บเท่าที่จำเป็น (Data Minimization)

- ฟอร์มขอ Permit เก็บเฉพาะข้อมูลที่จำเป็นต่อการอนุมัติ/ตรวจสอบหน้างาน
  (ชื่อ บริษัท อีเมล เบอร์โทร สถานที่ วันเวลา รายชื่อผู้ปฏิบัติงาน)
- **ห้าม**เพิ่มฟิลด์ที่ไม่จำเป็น (เช่น เลขบัตรประชาชน, วันเกิด, ที่อยู่บ้าน)
  โดยไม่มีเหตุผลทางธุรกิจที่ชัดเจนและแจ้งในประกาศความเป็นส่วนตัวก่อน
- endpoint ที่เปิดสาธารณะ (`doGet` action `status`) ส่งคืนเฉพาะฟิลด์ที่จำเป็น
  ต่อการตรวจสอบหน้างานเท่านั้น (ดู `handleGetStatus_` ใน .gs) — **ห้าม**เพิ่ม
  อีเมล เบอร์โทร รายชื่อผู้ปฏิบัติงานแบบเต็ม หรือเหตุผลไม่อนุมัติเข้าไปใน
  response นี้ เพราะใครก็ตามที่มีเลข Permit (เดาหรือรู้เลขได้) จะเรียกดูได้

### 2. แจ้งวัตถุประสงค์และขอความยินยอม (Notice & Consent)

- ผู้กรอกฟอร์มต้องเห็นประกาศ PDPA และติ๊กยินยอมก่อนกดส่งเสมอ (ดู checkbox
  `f_consent` ใน `request-permit.html` และการเช็ค `Consent_PDPA` ฝั่ง .gs)
- ถ้าเพิ่มช่องเก็บข้อมูลใหม่ที่กระทบวัตถุประสงค์เดิม (เช่น เก็บรูปใบหน้า,
  พิกัด GPS) ต้องปรับข้อความประกาศ PDPA ในฟอร์มให้ตรงกับของจริงด้วย

### 3. จำกัดสิทธิ์การเข้าถึง (Access Control)

| ข้อมูล | ใครเข้าถึงได้ | ควบคุมด้วย |
|---|---|---|
| Sheet เต็ม (ทุกคอลัมน์) | Safety Team เท่านั้น | Sheet sharing = **Restricted**, เพิ่มทีละอีเมล (Editor) |
| สถานะ Permit (`view.html`) | ใครก็ได้ที่รู้เลข Permit | ส่งเฉพาะฟิลด์จำเป็น ไม่มีช่องทางอ่านทั้ง Sheet |
| บันทึกเข้า-ออก | รปภ. ที่มี PIN ที่ Active=Yes เท่านั้น | ตรวจ PIN ทุกครั้งที่ scan (`findActiveGuardByPin_`) |
| Dashboard | ผู้ที่รู้รหัสผ่าน Dashboard | `Dashboard_Password` ใน Settings — **ต้องตั้งเอง** |
| ไฟล์ PDF/รูปถ่าย | ใครก็ได้ที่มีลิงก์ไฟล์นั้น | แชร์แบบ "Anyone with the link" **เฉพาะไฟล์เดียว** ไม่ใช่ทั้งโฟลเดอร์ |

**กฎเหล็ก: ห้ามเปลี่ยนการแชร์ Google Sheet เป็น "Anyone with the link" เด็ดขาด**
ไม่ว่าด้วยเหตุผลอะไร เพราะจะเปิดให้ใครก็ได้เห็นข้อมูลส่วนบุคคลทุกคอลัมน์ทันที

### 4. เก็บรักษาและลบข้อมูลตามความจำเป็น (Retention)

- ปัจจุบันระบบยังไม่มีการลบข้อมูลอัตโนมัติ — Safety Team ควรกำหนดรอบเวลา
  (เช่น ทุกสิ้นปี) ทบทวนและลบ/เก็บถาวรข้อมูล Permit ที่หมดความจำเป็นแล้ว
  ทั้งในแท็บ Sheet และไฟล์ PDF/รูปถ่ายใน Google Drive
- ถ้ามีการร้องขอใช้สิทธิ (เข้าถึง/แก้ไข/ลบข้อมูล) จากเจ้าของข้อมูล ให้ Safety
  Team เป็นผู้ดำเนินการใน Sheet โดยตรง (ระบบยังไม่มีฟังก์ชันอัตโนมัติสำหรับเรื่องนี้)

### เช็กลิสต์ก่อน commit/deploy การเปลี่ยนแปลงที่เกี่ยวข้องกับข้อมูล

- [ ] ฟิลด์ใหม่ที่เพิ่มเข้าไปจำเป็นต่อวัตถุประสงค์จริงหรือไม่
- [ ] ถ้า endpoint สาธารณะ (`doGet`/`doPost` ที่ไม่เช็คสิทธิ์) ส่งข้อมูลกลับ
      ตรวจว่าไม่หลุดฟิลด์อ่อนไหวเกินจำเป็น
- [ ] ถ้าเพิ่มการแชร์ไฟล์ Drive ใหม่ ตรวจว่าแชร์แบบ "ไฟล์เดียว" ไม่ใช่ทั้งโฟลเดอร์
- [ ] ถ้าข้อความประกาศ PDPA ในฟอร์มไม่ตรงกับข้อมูลที่เก็บจริงแล้ว ต้องแก้ข้อความ
      ให้ตรงกันก่อน merge
